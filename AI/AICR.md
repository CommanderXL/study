# AICR

先说结论：有价值，且要持续投入资源。

## 工程与模型

先回顾下 AICR 的几次架构升级：

### Prompt-Driven Workflow

最开始我们将 2024～2025 年 Q3 泛前端累计通报的 260 余起线上事故代码为样本，逐条借助 AI 分析事故成因、识别代码风险特征，在分析过程中引导模型提炼**可复用的问题识别规则**。经过人工校验和持续迭代后，这些规则被沉淀为 CR Rules，并与 CR Prompt、代码 Diff 一同注入模型上下文，辅助 LLM 在代码评审阶段识别同类风险。整体上，初期是打通从“历史事故分析”到“CR 规则提炼”，再到 AICR 的能力。

* 设计

Prompt Workflow 分为离线规则生产和在线代码评审两个阶段。

1. **离线规则生产**：以历史线上事故及对应代码为输入，通过预设 Prompt 引导 LLM 分析事故根因、定位风险代码模式，并提炼可复用的问题识别规则。规则经过人工校验、去重和修正后，统一沉淀到 CR Rules 中，作为后续代码评审的领域知识。

   历史事故 + 事故代码 + 规则提炼 Prompt → LLM → 人工校验 → CR Rules

2. **在线代码评审**：当代码变更触发 AICR 时，系统获取本次 Diff，将通用的 CR Prompt 与离线沉淀的 CR Rules 组合后，一次性注入同一个模型上下文。LLM 在理解代码变更的基础上，结合规则识别潜在风险，并输出问题描述、风险原因及修改建议。

   代码 Diff + CR Prompt + CR Rules → LLM → Review 结果

这一阶段整体采用单 Agent、单上下文的设计，工作流主要由 Prompt 驱动，没有对任务进行拆分，也没有在多个 Agent 之间进行上下文隔离或结果交叉校验。

* 问题

Prompt Workflow 的问题主要集中在工作流设计、模型能力和最终效果三个方面。

1. **工作流设计缺少任务拆分和工程约束**
   整个评审流程被集中在一个 Prompt、一次模型调用和一个对话 session 当中。工程能力和模型能力的职责不清，让 LLM 做了很多本应该放到工程侧做的事情，例如文件过滤等。模型既要理解代码变更，又要匹配规则、判断风险并组织输出，多种指令相互杂糅，指令之间的职责边界也不清晰，核心任务不突出（虽然通过 "重要" 这种关键词进行约束，但是效果并不好）。

2. **模型能力难以支撑复杂代码评审**
   当时模型的代码推理和上下文理解能力有限，AICR 更接近**基于 CR Rules 进行模式匹配，而不是结合调用链、业务语义和代码上下文完成完整推理**。受 128K/256K Context Window 限制，随时任务的持续进行，有效信息很容易被稀释，LLM 在 long-runing task 的指令遵循能力和输出稳定性明显下降。即便通过 Harness 约束，最终的效果也并不好。

3. CR 结果噪音多：
   模型容易泛化输出空值判断、异常捕获等“错误兜底类”建议，此类问题一度占全部评论的 50% 以上，但其中相当一部分没有结合实际业务上下文，难以直接指导开发者修改，也影响了团队对 AICR 结果的信任。


### Claude Code Agent SDK

<!-- Prompt-Driven Workflow 阶段暴露出的核心问题，并不是 Prompt 写得还不够复杂，而是我们将流程控制、确定性任务和代码推理都交给了 LLM。继续叠加 Prompt 和 CR Rules，只会进一步增加上下文负担，无法从根本上解决稳定性问题。因此，这一阶段的思考开始从“如何优化 Prompt”转向“如何重新划分工程与模型的职责”。 -->

同期 10 月，Claude Code Agent SDK 发布，提供了通过代码构建 Agent、调用工具以及编排和调度多个任务的能力。**这让 AICR 有机会引入更多工程手段：将原本不需要 LLM 参与的确定性工作交给代码完成，并通过工程流程约束 Agent 的输入、执行过程和输出结果，从而提升整个任务链路的可靠性**。

<!-- Agent SDK 的价值不只是更换模型调用方式，而是为 AICR 提供了一套将 LLM 纳入工程系统的运行时。工作流也由“Prompt 隐式驱动”逐步转变为“工程显式编排、Agent 负责推理”。 -->

基于前一阶段暴露的问题：

1. **划分工程与模型的边界**：
   相较于之前一个 prompt 里面描述了所有的任务及执行流程，本次迭代将代码 Diff 获取、文件过滤、输入裁剪和格式校验等目标明确、要求稳定的任务，逐步下沉到工程侧实现；LLM 更加聚焦于上下文分析及推理的任务，而把确定性更高的任务放到工程侧。

2. **从单一 Prompt 转向分阶段任务**
   不再要求一个 Prompt 同时完成代码理解、规则匹配、风险判断和报告生成，而是按照不同目标拆分任务。每个阶段只处理一类问题，使用相对独立的指令和上下文，减少复杂指令之间的相互干扰。

3. **过程产物写文件**
   不同任务之间不再只依赖同一个对话 Session 传递信息，而是通过结构化的中间结果进行衔接。这样既能控制单次任务的上下文规模，也能对每个阶段的结果进行校验、追踪和重新执行。（其实这里的设计就是 Harness Engineering）

这一阶段是 AICR 演进的重要转折点：Prompt 不再承担整个 Workflow，工程开始成为稳定性底座，模型则逐步回归到更擅长的理解、推理和判断任务。
